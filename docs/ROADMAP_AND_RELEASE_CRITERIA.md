# ReadNest 향후 개발 계획과 최종 배포 기준

## 현재 상태

ReadNest는 현재 Threads 링크 저장, 원문 추출, Gemini 요약, 아카이브/상세 화면 표시까지 MVP 핵심 흐름이 연결된 상태입니다.

핵심 흐름:

1. 사용자가 Threads 링크를 저장합니다.
2. 백엔드가 원문을 추출합니다.
3. Redis와 BullMQ summary queue에 요약 작업이 등록됩니다.
4. Worker가 Gemini API를 호출해 요약을 생성합니다.
5. 요약 결과가 DB에 저장됩니다.
6. 사용자는 홈, 아카이브, 상세 화면에서 요약을 확인합니다.

현재 단계의 중요한 목표는 기능을 무리하게 늘리는 것보다 실제 사용 중 발생하는 실패 케이스를 줄이고 사용자 경험을 안정화하는 것입니다.

## 앞으로의 개발 방향

다음 단계는 크게 다섯 가지입니다.

- 실제 기기에서 저장과 요약 흐름 안정화
- Threads 원문 추출 실패 케이스 감소
- 요약 결과를 더 읽기 좋은 구조로 분리
- 검색과 필터로 저장글 재탐색성 개선
- 소규모 사용자 배포를 위한 서버와 모바일 배포 환경 정리

## 최우선 작업

### 실제 기기 QA

실제 Android 기기와 iOS 시뮬레이터에서 다음 흐름을 점검합니다.

- 회원가입
- 로그인
- 토큰 유지
- 링크 저장
- 요약 상태 표시
- 요약 완료 후 상세 화면 반영
- 읽음 상태 변경
- 삭제
- 로그아웃
- 앱 재실행 후 로그인 유지
- 네트워크 불안정 상황에서 에러 표시

실제 기기 QA를 마치기 전에는 기능을 크게 늘리지 않는 것이 좋습니다.

### 요약 상태 UX 개선

요약은 큐 기반으로 비동기 처리되므로 저장 직후 결과가 바로 보이지 않을 수 있습니다.

필요한 상태:

- 저장됨
- 원문 추출 중
- 요약 대기 중
- 요약 중
- 요약 완료
- 요약 실패
- 원문 추출 실패
- 맥락 부족

목록과 상세 화면 모두에서 현재 상태를 확인할 수 있어야 합니다.

### 요약 실패 재시도

요약 실패는 실제 서비스에서 반드시 발생합니다.

고려해야 할 실패 원인:

- Gemini API 오류
- 원문 추출 실패
- 네트워크 타임아웃
- 본문이 너무 짧음
- 본문이 너무 김
- Threads 렌더링 실패

자동 재시도는 최대 1회 또는 2회로 제한하고, 이후에는 사용자가 수동으로 재시도하게 합니다.

### Threads 추출 실패 케이스 보강

ReadNest의 핵심 경쟁력은 Threads 글을 잘 가져오는 것입니다.

점검 케이스:

- 단일 Threads 글
- `1 of 9` 같은 번호형 Thread
- 댓글에 이어지는 Thread
- 이미지 포함 글
- 영상 포함 글
- 삭제된 글
- 비공개 또는 접근 제한 글
- 로그인 필요 글
- 본문이 매우 짧은 글
- 본문이 중복으로 추출되는 글
- 작성자 정보가 누락되는 글

추출 결과에는 원문 텍스트뿐 아니라 추출 신뢰도도 저장하는 것이 좋습니다.

## 요약 결과 개선

현재 `summary` 문자열 안에 구조화된 결과가 들어가 있다면 장기적으로 불편해질 수 있습니다.

권장 구조:

- 요약 유형
- 제목
- 한 줄 요약
- 핵심 요약
- 주요 포인트
- 태그
- 읽을 가치
- 주의점
- 맥락 상태
- 연속 글 상태
- 요약 신뢰도
- 원문 추출 신뢰도

이 구조를 별도 JSON 필드나 Summary 모델로 분리하면 프론트에서 각 영역을 따로 디자인하고 검색/필터에도 활용할 수 있습니다.

### 요약 유형별 UI

처음에는 UI를 완전히 다르게 만들 필요는 없습니다. 상세 화면에서 요약 유형 배지를 보여주고, 유형별로 중요한 섹션을 강조하는 정도면 충분합니다.

- 정보 정리형: 핵심 요약과 주요 포인트 중심
- 주장 분석형: 주장, 근거, 시나리오, 주의점 강조
- 학습 자료형: 핵심 개념, 학습 순서, 다시 볼 포인트
- 아이디어 저장형: 문제, 타깃, 적용 가능성
- 행동 추천형: 바로 할 행동과 주의점

## 검색과 필터

저장글이 30개를 넘기 시작하면 날짜별 아카이브만으로는 부족합니다.

우선순위:

- 제목 검색
- 요약 내용 검색
- 태그 필터
- 읽음 상태 필터
- 나중에 다시 보기 필터
- 기간 필터
- 출처 필터

초기에는 DB 단순 검색으로 시작하고, 저장글이 많아지면 전문 검색이나 벡터 검색을 검토합니다.

## 핵심 UX

### 오늘 읽을 글

ReadNest의 핵심 문제는 사용자가 글을 저장만 해두고 다시 읽지 않는 것입니다. 따라서 홈 화면 상단에는 전체 안 읽은 글 목록이 아니라, 부담 없이 다시 볼 만한 글을 큐레이션해서 보여주는 `오늘 읽을 글` 기능을 추가하는 것이 좋습니다.

기능 방향:

- 저장했지만 아직 읽지 않은 요약글 중 3개를 추천합니다.
- 전체 안 읽은 글을 상단에 모두 보여주지 않습니다.
- 사용자가 “할 일이 쌓였다”고 느끼지 않도록 가볍게 제안합니다.
- 홈 화면의 중심을 “오늘 저장한 글”보다 “오늘 다시 볼 만한 글”로 점진적으로 이동합니다.

추천 문구:

- 오늘 읽을 글
- 저장해두고 아직 읽지 않은 글 3개를 골랐어요
- 오늘은 이 3개만 가볍게 읽어보세요

카드에 표시할 정보:

- 제목
- 한 줄 요약
- 태그
- 저장 시점
- 읽음 표시 버튼

초기 추천 로직:

1. 요약 완료된 글
2. 안 읽은 글
3. 나중에 다시 보기로 표시된 글
4. 최근 저장된 글

MVP에서는 `요약 완료 + 안 읽음` 상태인 글 중 최근 저장순으로 최대 3개를 보여줍니다.

이후 개선:

- 나중에 다시 보기 우선
- 최근 저장글 우선
- 오래 방치된 글 일부 섞기
- 자주 보는 태그 우선
- 요약 신뢰도 높은 글 우선

상태별 홈 화면:

- 안 읽은 글이 있을 때: `오늘 읽을 글`과 추천 카드 3개, `안 읽은 글 전체 보기` 버튼
- 안 읽은 글이 없을 때: `오늘은 밀린 글이 없어요. 새로운 글을 저장하면 여기에서 추천해드릴게요`
- 요약 중인 글이 있을 때: `요약 중인 글 N개. 요약이 끝나면 오늘 읽을 글에 추가돼요`

권장 홈 화면 순서:

1. 오늘 읽을 글
2. 요약 중인 글
3. 오늘 저장한 글
4. 이번 주 요약

포트폴리오 설명:

> 사용자가 저장만 하고 다시 읽지 않는 문제를 줄이기 위해, 요약이 완료되었지만 아직 읽지 않은 글을 홈 화면 상단에 노출하는 Today Queue를 구현했습니다. 초기 추천 로직은 요약 완료 여부와 읽음 상태를 기준으로 구성하고, 이후 태그와 저장 시점을 반영할 수 있도록 확장 가능한 구조로 설계했습니다.

향후 홈 전용 API:

- 오늘 읽을 글
- 요약 중인 글
- 오늘 저장한 글
- 안 읽은 글 수
- 이번 주 저장 수

오늘 읽을 글 조건:

- 사용자 본인의 글
- 요약 완료
- 읽음 상태가 안 읽음 또는 나중에 다시 보기
- 삭제되지 않음
- 원문 링크 존재
- 최대 3개

정렬:

- 나중에 다시 보기 우선
- 최근 저장글 우선

### OS 공유 저장

ReadNest가 링크 붙여넣기보다 편하려면 OS 공유 저장이 필요합니다.

목표 흐름:

1. Threads에서 공유 버튼을 누릅니다.
2. ReadNest를 선택합니다.
3. 링크가 자동 저장됩니다.
4. 저장 완료 메시지가 보입니다.
5. 요약은 백그라운드에서 진행됩니다.

Expo Go에서는 공유 시트나 딥링크 테스트에 한계가 있을 수 있으므로 개발 빌드 또는 EAS Build 구성을 검토합니다.

### 요약 복사와 공유

우선 구현할 기능:

- 한 줄 요약 복사
- 전체 요약 복사
- 원문 링크 복사
- 요약과 원문 링크 함께 복사
- 요약 공유하기

공유 결과에는 원문 출처와 링크가 반드시 포함되어야 합니다.

### 나중에 다시 보기

읽음/안 읽음만으로는 부족할 수 있으므로 `READ_LATER` 상태는 유지합니다.

필터:

- 전체
- 안 읽음
- 읽음
- 나중에 다시 보기

## 백엔드 안정화

### 에러 처리

사용자에게 보여줄 에러와 서버 내부 에러를 분리합니다.

사용자 메시지 예시:

- 원문을 가져오지 못했어요
- 요약에 실패했어요
- 잠시 후 다시 시도해 주세요
- 이미 저장된 글이에요

서버 로그 예시:

- Playwright timeout
- Gemini API error
- Redis connection error
- Prisma query error

### Worker 운영 정리

점검 항목:

- 동시 작업 수
- 재시도 횟수
- 실패 job 보관
- 완료 job 보관
- timeout
- Redis 연결 실패 처리
- 서버 재시작 후 job 유지
- Worker가 꺼져 있을 때 저장글 상태

### Playwright 운영 설정

운영 기준:

- 페이지 로딩 timeout
- 추출 최대 대기 시간
- 본문 최대 길이
- 브라우저 인스턴스 재사용 여부
- 동시 추출 제한
- 실패 시 fallback 추출

운영 서버에서는 Playwright 실행 환경이 까다로울 수 있으므로 배포 환경에서 반드시 테스트합니다.

### API 사용량 제한

Gemini API 비용과 남용을 막기 위해 제한이 필요합니다.

기본 제한:

- 사용자별 하루 저장 제한
- 사용자별 하루 요약 제한
- 동일 URL 재요약 제한
- 요약 재시도 제한
- 너무 긴 본문 요약 제한

초기에는 관리자 또는 테스트 계정만 요약 가능하게 해도 좋습니다.

## 데이터 모델 개선

요약 결과는 문자열 하나로만 저장하지 않는 것이 좋습니다.

권장 방식은 Summary 모델 분리입니다.

장점:

- 요약 재생성 이력을 남길 수 있습니다.
- 모델별 요약 결과를 비교할 수 있습니다.
- 요약 상태를 따로 관리할 수 있습니다.
- 추후 고급 요약과 간단 요약을 분리할 수 있습니다.

MVP 속도를 우선하면 JSON 필드부터 시작해도 됩니다.

## ThreadGroup 통합 요약

다음 단계에서 필요한 기능:

- 같은 Thread에 속한 파트를 하나의 그룹으로 묶기
- 각 파트의 번호와 내용 저장
- 전체 파트 수와 수집된 파트 수 비교
- 전체가 수집되면 통합 요약 생성
- 일부만 수집되면 부분 요약으로 표시

상세 화면 표시:

- 전체 포함
- 일부 포함
- 수집된 파트 수
- 누락 가능성
- 요약 신뢰도

## 보안 기준

최소 배포 전 기준:

- API key는 프론트에 노출하지 않습니다.
- JWT secret은 환경변수로 관리합니다.
- Gemini API key는 환경변수로 관리합니다.
- 사용자는 본인 저장글만 조회할 수 있어야 합니다.
- 다른 사용자의 content id 접근은 차단합니다.
- 비밀번호는 해시 처리합니다.
- 삭제 요청은 본인 데이터만 가능해야 합니다.
- 운영 DB 접속 정보는 저장소에 올리지 않습니다.
- 환경변수 파일은 git에 포함하지 않습니다.

- 읽음, 안 읽음, 나중에 다시 보기 상태를 변경할 수 있습니다.
- 저장글을 삭제할 수 있습니다.
- 요약 실패 시 재시도할 수 있습니다.
- 검색 또는 기본 필터가 동작합니다.

### 요약 품질 기준

- 제목이 원문 내용을 벗어나지 않습니다.
- 한 줄 요약이 글의 핵심을 담습니다.
- 핵심 요약은 3문장 안에서 의미가 자연스럽습니다.
- 주요 포인트는 중복 없이 3개에서 5개로 정리됩니다.
- 태그는 나중에 찾기 쉬운 단어로 생성됩니다.
- 투자, 건강, 법률, 정치 주제에는 주의점이 표시됩니다.
- 연속 Thread는 전체 포함 여부가 표시됩니다.
- 일부만 수집된 글은 맥락 부족이나 부분 요약으로 표시됩니다.
- 원문에 없는 사실을 만들어내지 않습니다.

### 안정성 기준

- 앱이 저장 과정에서 자주 멈추지 않습니다.
- 저장 완료 피드백이 빠르게 표시됩니다.
- 요약은 백그라운드에서 처리됩니다.
- 요약 중 상태가 명확하게 보입니다.
- 서버 에러가 발생해도 앱이 터지지 않습니다.
- 네트워크 오류가 사용자에게 적절히 표시됩니다.
- Worker가 꺼져 있을 때도 데이터가 손상되지 않습니다.
- 재시작 후에도 큐 작업이 이어집니다.

### 비용 기준

- 사용자별 하루 요약 제한이 있습니다.
- 같은 URL의 불필요한 재요약을 막습니다.
- 요약 재시도 횟수를 제한합니다.
- Gemini API 사용량을 기록합니다.
- 본문 길이 제한이 있습니다.
- 테스트 사용자 수를 제한할 수 있습니다.

### 운영 기준

- 서버 로그를 확인할 수 있습니다.
- 요약 실패 로그를 확인할 수 있습니다.
- API 배포 환경이 정리되어 있습니다.
- DB 백업 또는 복구 방법을 알고 있습니다.
- Redis와 Worker 실행 방법이 문서화되어 있습니다.
- 에러 발생 시 임시 대응 방법이 있습니다.

### 사용자 테스트 기준

- 최소 5명에게 테스트를 진행합니다.
- 각 사용자가 직접 Threads 글을 저장해봅니다.
- 각 사용자가 최소 5개 이상의 글을 저장해봅니다.
- 저장 실패 사례를 수집합니다.
- 요약이 유용한지 피드백을 받습니다.
- 사용자가 다시 앱에 들어와 요약을 확인하는지 확인합니다.
- 어떤 화면이 불편한지 기록합니다.
